## 实习课题任务书

基于 StockBench 的金融 LLM Agent 研究
课题周期：2周 | 方向：大模型 × 金融

## 1. 背景

StockBench 是清华大学与北京邮电大学联合发布的金融 LLM Agent Benchmark（2025年），用于评估大语言模型在真实股票交易环境中的表现。Agent 每个交易日接收价格、基本面、新闻三类市场信号，输出买入、卖出或持有决策，评估指标为量化金融标准指标。

论文已测试 GPT-5、Claude 4、Qwen3、Kimi-K2 等主流模型作为 baseline，核心发现是：几乎所有模型的 Sortino Ratio 都高于 Buy & Hold 基准，但彼此之间差异有限。这说明当前 agent 架构存在明显的改进空间，是本课题的核心切入点。

- 论文：StockBench: Can LLM Agents Trade Stocks Profitably In Real-world Markets?（arXiv:2510.02209）
- GitHub：https://github.com/ChenYXxxx/stockbench
- 数据：DJIA 前 20 只股票，通过 Polygon.io 和 Finnhub 免费 API 获取

## 2. 课题目标

在 StockBench 框架上，通过改进 agent 的架构、信号处理或决策机制，使量化评估指标相对于原版 baseline 有显著提升，并给出有实验支撑的分析。

> 注意：课题重点不是刷指标数字，而是理解为什么现有 agent 表现有限，并提出有针对性的改进。分析过程和实验设计本身同等重要。

## 3. 评估规范（所有人统一）

为保证实习生之间结果可比，以下评估规范强制执行。

### 3.1 数据与环境

- 股票池：DJIA 前 20 只股票（同原论文）
- 回测时间窗口：March 3 - June 30, 2025 (82 trading days)，使用 yfinance 拉取历史数据，避免 Polygon 免费 tier rate limit 问题
- 初始资金：$100,000
- LLM API：使用统一分配的 API key，模型版本固定（待定）

### 3.2 评估指标

| 指标             | 说明                           | 角色     |
| ---------------- | ------------------------------ | -------- |
| Sortino Ratio    | 下行风险调整收益，对亏损更敏感 | 主指标   |
| Total Return (%) | 相对 Buy & Hold 的超额收益     | 辅助指标 |
| Max Drawdown (%) | 最大回撤幅度                   | 辅助指标 |

### 3.3 统计要求

- 每个实验方案重复运行 5 次，汇报均值 ± 标准差
- 必须包含 Buy & Hold 作为 baseline 对照
- 改进幅度以相对 baseline 的变化量来衡量，而非绝对数值
- 如果改进在标准差范围内，需说明原因并讨论统计显著性

## 5. 可选探索方向

以下方向供参考，不限于此。鼓励结合自己的判断提出不同的切入点。

### 方向 A：决策机制改进

原版框架是单步 prompt → 决策。可以探索带 reflection 的两阶段决策（先生成市场分析，再基于分析决策）、引入风险审查 agent 做二次判断等方式，研究是否能在降低 Max Drawdown 的同时维持收益。

### 方向 B：信号消融与重组

框架同时输入价格、基本面、新闻三类信号，但各信号的实际贡献未被量化。可以系统地做消融实验（控制 agent 能看到的信号子集），分析不同信号在不同市场状态下的贡献，并基于此设计自适应的信号选择策略。

### 方向 C：记忆与上下文管理

Agent 目前对历史决策的记忆方式比较简单。可以研究不同的上下文窗口策略（长 vs 短历史、选择性记忆 vs 全量历史），或引入对历史错误决策的显式反思机制，分析对决策质量的影响。

### 方向 D：自定义探索

如果你在阅读代码和论文后发现了更有意思的切入点，欢迎自行设计方向。建议在第 3–4 天和导师对齐后再开始实现。

## 6. 产出要求

### 6.1 代码

- 基于原版 StockBench 框架，改动清晰、有注释
- 包含完整的实验复现说明（README 或注释）
- 结果自动保存为结构化格式（CSV 或 JSON），方便统一汇总对比

### 6.2 报告

- 背景与动机：为什么选择这个改进方向，分析了哪些 baseline 的问题
- 方法描述：改动了什么，为什么这样改
- 实验结果：按统计要求汇报指标，含均值 ± 标准差
- 分析与结论：改进是否有效，为什么，局限性在哪里

长度不限，清晰即可。

### 6.3 汇报

- 时长：15 分钟展示 + 10 分钟 Q&A
- 建议包含 demo（实际跑一段回测展示 agent 决策过程）

## 7. 资源与注意事项

### 7.1 API 与数据

- Polygon.io 免费 API：5次/分钟，建议用 yfinance 拉历史数据替代，避免限速
- Finnhub 免费 API：60次/分钟，用于获取新闻数据，够用
- LLM API key 另行分配，有用量限制，注意控制调试成本

### 7.2 工具建议

- 鼓励使用 vibe coding 工具（Cursor、Claude Code 等）加速开发，但需理解自己写出的代码
- 实验结果用脚本自动汇总，不要手动整理数据

如果在环境配置或框架理解上遇到问题，第 1–2 天内尽早反馈，不要卡在环境问题上消耗过多时间。
